Section 1

Executive Summary

The Agent Tracker application organizes the teams, contacts, locations, and scheduling that support the State Sponsored Life Insurance Program (SSLI) and RGLI. This document describes Version 2.0 of the data model — an evolution of the application that expands its capabilities significantly while preserving everything that made the original work.

🚀

A Strong Foundation, Taken Further

The original Agent Tracker established the core concepts that this program depends on — Briefings, Units, Contacts, and Users. Version 2.0 takes those same concepts and deepens the relationships between them, unlocking a new tier of reporting, visibility, and operational intelligence that the program is now ready to take advantage of.

What This Version Adds

The Version 2.0 data model expands the application in four meaningful ways:

🔗 Richer Relationships

Contacts, Units, and Briefings can now be linked to each other in true many-to-many relationships — reflecting how the real world actually works. A contact can belong to multiple units; a briefing can span multiple units; an agent can serve multiple roles.

🏢 Deeper Unit Intelligence

Units now carry their own briefing history — automatically tracking when they were last briefed, when the next briefing is scheduled, and whether they are current or overdue. No manual tracking required.

👤 Flexible User Roles

The same person can serve as an agent at one unit and a manager at another simultaneously. Role permissions are clearly defined and automatically cascaded — Directors are always Managers; Managers are always Agents.

📊 Reporting at Scale

Because every relationship is stored explicitly, the application can now answer multi-dimensional operational questions — filtering by agent, state, program, date range, unit status, and more — all at once.

Why this matters for leadership: The Version 2.0 model was designed with reporting in mind from the ground up. Managers and directors will be able to see the full picture of program activity — which units are overdue, which agents are most productive, which states have coverage gaps — without manual spreadsheets or data exports. See Section 6 for the full list of reporting capabilities this unlocks.

Section 3

Relationships Explained

Every line in the diagram above represents a real-world relationship that the application tracks. Here is what each one means in plain terms — and why it matters operationally.

Contacts ↔ Units

A military contact can be associated with more than one unit — for example, a service member who transfers or holds a role that spans installations. Conversely, a unit will have many contacts over time. The ContactUnits table stores every pairing, giving agents a complete picture of who they know at each location.

A contact's primary address comes from their unit. If a contact is not tied to a unit, they can have their own alternate address stored directly on their record — ensuring no one falls through the cracks.

Briefings ↔ Units

A briefing is associated with the unit(s) where it takes place. The BriefingUnits table records which units are connected to each briefing, and the unit's program designation (SSLI or RGLI) flows through to the briefing automatically — eliminating the need for redundant data entry and the errors that come with it.

Briefings ↔ Contacts

A briefing can involve many contacts (everyone in the room), and a contact can attend many briefings throughout their career. The BriefingContacts table preserves the full attendance record for every briefing, creating a permanent, queryable history of engagement that grows more valuable over time.

Units ↔ Users (Agents)

An agent can be assigned to cover more than one unit, and a unit can have more than one agent working it. The UnitAgents table stores these assignments, making it easy to see exactly who is responsible for each unit and which units each agent is covering.

Units ↔ Users (Managers)

Separately from agent assignments, managers are linked to the units they oversee through the UnitManagers table. Critically, the same user can appear as an agent at one unit and as a manager at another — this design supports the real-world complexity of how your team operates.

Users ↔ States (Licenses)

Agents are licensed to operate in specific states. The UserStateLicenses table records every state in which each agent holds a license. This makes it straightforward to match agents to units, verify coverage, and ensure compliance — all from within the app.

States (Reference Table)

The States table is a shared reference list of all 50 US states used as a consistent, controlled dropdown wherever a state field appears — in unit addresses, contact addresses, briefing locations, user home addresses, and license records. This single reference ensures that "Texas" is always "Texas" across every table, making state-level filtering and reporting perfectly reliable.

Section 5

Junction Tables

Junction tables are what make many-to-many relationships possible. Each one exists to represent a connection that neither side can fully describe on its own — for example, an agent covering multiple units, or a contact attending multiple briefings. These tables are what give Version 2.0 its reporting depth.

Why junction tables are so powerful for reporting: Because every relationship is stored as an explicit record, the application can answer precise operational questions — "which agents cover units in Virginia?", "which contacts attended the most briefings last year?", "which units have no agent assigned?" — instantly, without any manual data compilation.

ContactUnits Junction

Links contacts to the units they are associated with. One contact can be connected to many units; one unit can have many contacts. This table is the foundation for contact-level reporting across locations.

ColumnTypeNotes
ContactIdRef → Contacts
UnitIdRef → Units

BriefingUnits Junction

Links briefings to the units they are associated with. One briefing can span multiple units; one unit accumulates a rich briefing history over time. Program and branch data roll up through this table automatically.

ColumnTypeNotes
BriefingIdRef → Briefings
UnitIdRef → Units

BriefingContacts Junction

The attendance record — which contacts were present at which briefings. Every entry here is a permanent piece of relationship history that can be queried, filtered, and reported on for years to come.

ColumnTypeNotes
BriefingIdRef → Briefings
ContactIdRef → Contacts

UnitAgents Junction

Records which agents are assigned to cover which units. Enables full visibility into coverage — who is responsible for each unit, and how many units each agent carries.

ColumnTypeNotes

UnitManagers Junction

Records which managers oversee which units. Kept intentionally separate from UnitAgents so that a user can serve as an agent at one unit and a manager at another at the same time — reflecting how teams actually operate in the field.

ColumnTypeNotes
UnitIdRef → Units

Tracks the states in which each agent holds an active license. Supports compliance, coverage analysis, and smart assignment recommendations — ensuring agents are always matched to units where they are authorized to operate.

Section 6

Reporting Capabilities

This is where the Version 2.0 data model truly shines. Because every relationship between agents, units, contacts, and briefings is now stored as a distinct, queryable record, the application can answer operational questions at a depth and speed that simply was not possible before — without spreadsheets, without manual data pulls, and without waiting.

The compounding effect of connected data: Each of the reports below is powerful on its own. What makes this model exceptional is that they can all be combined and cross-filtered simultaneously. "Show me all units in Texas assigned to Agent Smith that are overdue, and list the key contacts at each one" — that query spans five tables and returns in seconds because of how this model is structured.
⚠️
Unit Health at a Glance

Every unit in the system displays its current status automatically — Overdue, Need Scheduling, Is Current, or Never Briefed. No manual tracking. Always up to date. Filter by state, branch, program, or manager in seconds.

📊

How many briefings has each agent completed this quarter? How many applications obtained? Which of their units are falling behind? All visible in one place, per agent, without any manual compilation.

📈
Applications by Program

Total applications obtained for SSLI vs. RGLI — by agent, by state, by time period, by branch of service. Because Program lives on the Unit and flows to every Briefing, this report is always accurate.

📅
Scheduling Pipeline

Which units have a briefing coming up? Which have nothing scheduled and are approaching the overdue threshold? Proactive scheduling visibility, built in.

🗺️
Geographic Coverage Analysis

Which states have the most active units? Which agents are licensed in those states? Where are the coverage gaps? Program leadership can see the full national picture at any time.

🤝
Contact Engagement History

Which contacts have attended briefings in the past year? Which key contacts at a given unit have never been to one? Relationship depth is now measurable and trackable.

🏗️
Manager Oversight & Roll-Up

Every manager can see the units they oversee and the agents under them. Directors get a rolled-up view across all managers. The hierarchy is automatic — no manual aggregation needed.

✅
Compliance & Licensing

With state licenses stored per agent, the app can flag any situation where an agent is assigned to a unit in a state where they are not yet licensed. Auditable, proactive, and automatic.

📉
Trend Analysis Over Time

Because every briefing and every contact attendance is a permanent record, trend analysis across any time period is always available — month over month, quarter over quarter, year over year.

Section 7

Key Business Rules

Several fields in the application calculate their values automatically based on the data already in the system. This section documents those rules clearly so every stakeholder understands how the app reaches its conclusions.

User Role Transitivity

Roles follow a clean hierarchy that cascades automatically when a higher role is assigned:

  • All users are Agents. IsAgent is always TRUE — no manual entry needed.
  • If IsDirector = TRUE, then IsManager is automatically TRUE.
  • IsAdmin is independent — it grants system administration access and can be combined with any other role.
  • A user can serve as an Agent at one unit and a Manager at a different unit at the same time. These are tracked as separate, clean records in UnitAgents and UnitManagers.

Unit Briefed Status (Automatic)

Each unit's briefing status is calculated automatically — no one has to update it manually. The logic is:

StatusConditionWhat It Means
Is Current1 to 12 months since last briefUnit has been briefed within the past year — in good standing
Need Scheduling12 to 18 months since last briefApproaching the threshold — should be scheduled soon
OverdueMore than 18 months since last briefPast the required window — needs immediate attention
Never BriefedNo LastBriefedDate on recordUnit has no briefing history — a priority for initial outreach

LastBriefedDate and ScheduledBriefDate

  • ScheduledBriefDate on a unit is drawn automatically from any linked Briefing with a future date and a status of "Scheduled."
  • When a ScheduledBriefDate passes, it automatically becomes the LastBriefedDate and ScheduledBriefDate clears — ready for the next briefing to be entered.
  • Both fields are virtual and calculated — agents never have to update them manually.

Program Inheritance

  • Program (SSLI or RGLI) is set once at the Unit level.
  • Every Briefing connected to that unit inherits its Program automatically via the BriefingUnits relationship.
  • This eliminates duplicate entry and ensures that all program-level reporting is always consistent with the unit's classification.

Contact Address Logic

  • A contact's address is drawn from the unit(s) they are linked to via ContactUnits — no separate entry needed in the common case.
  • If a contact has no unit association, or has a personally different address, the AltAddress fields on the Contact record are used instead.
  • The same AltAddress pattern applies to Briefings held at a location other than the unit's primary address.
